**담당:** 강대리(백엔드)  
**시각:** 2026-07-06 23:01 JST  
**원칙:** 운영 DB 무변경, Secret 비노출

## 결론

**FAIL — 운영 Supabase 정상 여부를 실제 응답으로 확인하지 못했다.**

이는 운영 DB 장애 판정이 아니다. 현재 실행 환경에서 Supabase 호스트 DNS 해석이 실패하고, SQL 접속 URL·Management API Token·연결된 브라우저가 없어 운영 카탈로그와 Data API에 도달하지 못한 **검증 실패**다.

| 검증 항목 | 판정 | 실행 결과 |

## 실제 수행 내용

1. 최신 DB 보고서, `db/schema.sql`, `db/verify_schema.sql`, 웹 RPC 호출부, 파이프라인 업서트 경로를 대조했다.
2. `.env`에 Data API용 `SUPABASE_URL`·`SUPABASE_SECRET_KEY`가 있고 DB 직접 연결 정보는 없음을 값 비노출로 확인했다.
3. service_role 자격으로 임베딩 실데이터 1건 조회 후 저장된 768차원 벡터를 넣는 4인자 `match_filings` 자기검색을 시도했다. 기대 조건은 HTTP 200, 동일 기업, similarity 0.99 이상이었다.
4. 유효한 HTTPS URL·키 존재까지 확인했지만 운영 호스트 DNS 해석 실패로 Data API 응답을 받지 못했다.
5. 인앱 브라우저 연결 목록도 0개여서 로그인된 Supabase SQL Editor 경로가 없었다.
6. 운영 DB에는 INSERT·UPDATE·DELETE·DDL을 전혀 실행하지 않았다.

## 완료 판정

이번 환경에서 가능한 읽기 검증을 모두 수행했으나 운영 증거를 확보하지 못했으므로 **작업 결과는 FAIL로 종료**한다. PASS 전환 조건은 네트워크 가능한 환경에서 다음 결과를 확보하는 것이다.

```bash
psql "$SUPABASE_DB_URL" -v ON_ERROR_STOP=1 -f db/verify_schema.sql
# 마지막 결과: PASS
```

이후 service_role Data API에서 실데이터 SELECT와 4인자 RPC 자기검색이 모두 HTTP 200으로 응답해야 한다.
