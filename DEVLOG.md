## 아이디어의 확장 과정

처음 아이디어는 한 줄이었다: **"관심 종목 공시를 매일 아침 요약해서 메일로 받자."**

그런데 만들다 보니 각 단계가 다음 질문을 낳았다.

1. 공시를 매일 수집·요약한다 → *"지나간 공시를 다시 찾고 싶으면?"* → **RAG 검색** (공시 원문을 벡터DB에 쌓기 시작)
2. 검색 데이터가 쌓였다 → *"메일 말고 아무 때나 질문하고 싶다"* → **Q&A 웹앱** (질문 → 근거 공시와 함께 AI 답변)
3. 웹이 생겼다 → *"내 종목은 나만 아는데, 사이트는 왜 모두에게 같지?"* → **회원 + 포트폴리오** (로그인, 보유 종목·평단가 등록, 실시간 수익률)
4. 내 종목을 등록했다 → *"그럼 수집도 알림도 내 종목 기준이어야지"* → **개인화** (보유 종목 자동 수집, 회원별 맞춤 알림, 중요 공시 ⚠️ 감지)
5. 포트폴리오에 미국 주식이 있다 → *"달러 종목도 넣고 싶다"* → **다중 시장** (미국·일본 시세, 환율 자동 환산)

돌아보면 "기능을 추가한" 게 아니라, **쓰는 사람 입장에서 다음에 막히는 지점을 계속 뚫은 것**에 가깝다.

## Phase 1 — 파이프라인 (7/4)

**만든 것:** DART 공시 수집 → Gemini 요약 → Gmail 발송 → GitHub Pages 대시보드. GitHub Actions cron으로 매일 아침 자동 실행.

**왜 이 구조인가:**
- **GitHub Actions를 서버로 쓴 이유** — public 저장소는 실행 시간 무제한 무료. 서버 비용 0원으로 매일 도는 배치가 생기고, CI/CD 경험 자체가 포트폴리오가 된다.
- **Gemini flash-lite를 고른 이유** — 무료 티어로 하루 8종목 요약이 충분히 돌아간다. 쿼터(429)에 걸려본 뒤 재시도 + 종목별 실패 격리를 넣었다. **한 종목의 실패가 전체 브리핑을 죽이면 안 된다**는 원칙은 이후 모든 단계에 적용됐다.

**부딪힌 것:** cron은 UTC 기준이라 요일 계산이 밀려 금요일 공시가 누락되는 버그. 시간대 문제는 "돌아가는 걸 확인했다"와 "매일 돌아간다"가 다르다는 걸 가르쳐줬다.

## Phase 2 — RAG와 Q&A 웹 (7/4 ~ 7/6)

**만든 것:** 공시 원문을 청크로 쪼개 Gemini 임베딩 → Supabase(pgvector) 저장. Next.js 웹에서 질문하면 유사도 검색(match_filings) → 근거와 함께 답변. Vercel 배포.

**왜 이 구조인가:**
- **요약과 인덱싱을 분리한 이유** — 요약은 "그날의 완성품"(매일 새로 생성), 인덱싱은 "누적 자산"(중복 스킵). 성격이 다른 데이터는 수명 주기도 다르게 관리한다.
- **답변 근거로 요약본이 아닌 원문 청크를 쓰는 이유** — "발행 규모 얼마?" 같은 세부 수치는 요약 과정에서 사라진다. 신뢰가 필요한 답변은 원문에서 나와야 한다.
- **Next.js API Route가 필요했던 이유** — 정적 페이지는 코드가 전부 공개되어 DB secret key를 숨길 곳이 없다. 서버가 필요해진 최초의 순간.

**보안 결정:** anon/authenticated 권한 전면 차단, secret key는 서버 전용, 공개 질문 API에는 Upstash 기반 rate limit. "무료 서비스라도 열어두면 내 Gemini 쿼터는 남이 쓴다."

## Phase 3 — 회원과 포트폴리오 (7/9 ~ 7/10)

**만든 것:** Supabase Auth 로그인, 보유 종목(수량·평단가) 등록, 실시간 시세 기반 수익률·비중 도넛차트.

**왜 이 구조인가:**
- **supabase-js를 안 쓰고 GoTrue/PostgREST를 REST로 직접 호출** — 기존 코드베이스가 "클라이언트에 키 노출 금지, NEXT_PUBLIC 금지" 원칙이었다. 세션은 httpOnly 쿠키로만 유지. 라이브러리 편의보다 보안 원칙의 일관성을 택했다.
- **RLS(행 수준 보안)에 접근 제어를 맡긴 이유** — 서버 코드가 user_id를 다루지 않는다. "자기 행만 보인다"를 DB가 강제하면 서버 버그가 나도 남의 데이터에 닿을 수 없다.
- **시세를 프록시(/api/quotes) 한 파일에 격리한 이유** — 비공식 API(네이버)는 언제든 바뀔 수 있다. 나중에 공식 API(토스증권 등)로 갈아탈 때 이 파일 하나만 교체하면 되는 구조.

**부딪힌 것 (42501):** holdings 테이블을 만들고도 계속 502. 원인은 RLS가 아니라 **테이블 GRANT** — Phase 2에서 보안 강화를 위해 기본 권한을 전부 회수해둔 것이 새 테이블에도 적용된 것이다. 과거의 보안 결정이 미래의 나를 막은 사례. 에러 메시지에 원인 코드를 그대로 노출하도록 고치고 나서야 5분 만에 잡았다. → **교훈: 디버깅 가능한 에러 메시지가 가장 싼 투자다.**

## Phase 3.5 — 개인화와 알림 (7/10)

**만든 것:** 보유 종목이 자동으로 수집 대상에 합류, 미등록 종목은 웹에서 "지금 수집" 버튼(GitHub Actions 원격 실행), 회원별 맞춤 아침 메일(자기 종목 공시만), 중요 공시(유상증자·합병 등 19종) 제목 ⚠️ 표시.

**왜 이 구조인가:**
- **온디맨드 수집을 동기 처리하지 않고 Actions로 넘긴 이유** — 수집+임베딩은 수 분 걸린다. 웹 함수 타임아웃 안에서 처리하는 대신, 기존 파이프라인 코드와 secrets를 그대로 재사용하는 workflow_dispatch가 구현도 안전도 낫다. 웹은 20초 간격 폴링으로 완료를 감지해 자동 재검색한다.
- **"공시 없음" 메일을 따로 보내지 않는 이유** — 빈 메일이 반복되면 스팸이 되고, 정작 ⚠️ 알림도 안 읽게 된다. 대신 메일 하단에 "오늘 공시가 없었던 내 종목" 한 줄로 절충했다.
- **watchlist를 필수에서 선택으로 바꾼 이유** — 회원의 보유 종목이 수집원이 된 순간, 하드코딩된 목록은 부트스트랩용일 뿐이다. 데이터의 주인이 운영자에서 사용자로 넘어간 전환점.

**부딪힌 것:** "skt"로 검색하면 수집 실패 — DART는 정확한 법인명만 받는다. 별칭 사전을 만들다가 접었다(확장 불가). 대신 **수집 요청 시 서버가 검색 API로 정식 종목명을 먼저 해석**하는 일반해로 바꿨다. → **교훈: 개별 케이스를 때우는 코드보다, 케이스가 생기지 않는 구조.**

## Phase 4 — 다중 시장 (7/10)

**만든 것:** 미국(나스닥·NYSE·AMEX)·일본(도쿄) 주식 등록, 달러·엔 시세, USD/JPY 환율 자동 조회, 모든 합산은 원화로 통일.

**왜 이 구조인가:**
- **수익률은 현지 통화로, 평가금액은 원화로** — 수익률에 환율이 섞이면 "주가는 올랐는데 환율로 마이너스"가 한 숫자에 뭉개진다. 분리해야 정보다.
- **환율 실패 시 화면을 깨지 않고 "—"로** — 시세는 오는데 환율만 실패할 수 있다. 그 경우에도 현지 통화 수익률은 보여주고, 원화 환산 칸만 비운다.

**부딪힌 것 (이번 개발의 가장 큰 교훈):** 해외 시세·검색·환율 엔드포인트를 전부 추측으로 넣었고, 전부 빗나갔다. 해결법은 단순했다 — **브라우저로 실제 URL을 열어 JSON을 눈으로 확인**하고 그 형식에 맞춰 다시 짰다. 그 과정에서 발견한 것: 국내용이라 생각했던 자동완성 API가 사실 전 세계 종목을 반환하고 있었다(nationCode로 거르기만 하면 됐다). → **교훈: 비공식 API는 문서가 없으므로 응답 실물이 유일한 스펙이다. 추측으로 짠 코드는 반드시 실측으로 검증한다.**

## 반복해서 나타난 원칙들

1. **실패 격리** — 종목 하나, 회원 하나, API 하나의 실패가 전체를 멈추지 않게. (요약 실패 시 목록 표시, 알림 발송 개별 try, 시세 실패 시 "—")
2. **비밀은 서버에만** — NEXT_PUBLIC 금지, httpOnly 쿠키, RLS. 클라이언트는 처음부터 신뢰하지 않는다.
3. **교체 가능성 격리** — 바뀔 수 있는 외부 의존(네이버 시세, Gemini 모델명)은 파일/환경변수 하나에 가둔다.
4. **에러는 디버깅 가능하게** — "실패했습니다"가 아니라 "(오류 42501) permission denied"까지. 원인 코드가 보이면 해결은 몇 분이다.
5. **개별 해킹보다 일반해** — 별칭 사전 대신 이름 해석, 종목 추가 대신 사용자 데이터 기반 수집.
