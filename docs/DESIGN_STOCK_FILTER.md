## 0. 이 문서의 위치

FE가 이미 자유 입력형 종목 필터를 1차 구현했다(`ask-panel.tsx`의 `company-filter`). 이 문서는
그 구현의 **설계 근거를 사후 정리하고, 정합성 점검 기준과 남은 개선점을 확정**하기 위한 참조
사양이다. 새 컴포넌트 재설계가 아니라, 현재 구현을 기준으로 "무엇이 맞고, 무엇을 보완해야
하는지"를 FE가 판단할 수 있게 하는 것이 목적이다.

각 섹션 끝의 **`▷ 현재 구현`** 표시는 지금 코드 상태를, **`▷ 보완`** 표시는 FE가 추가로
반영하면 좋은 항목을 뜻한다.

---

## 1. 설계 원칙

1. **선택적·비강제.** 종목 필터는 필수가 아니다. 비워 두면 전체 공시에서 검색한다. 필터를
   비운 상태가 기본값이며, 사용자가 의도적으로 좁힐 때만 값을 넣는다.
2. **질문이 주(主), 필터가 보조.** 화면의 주인공은 질문 입력이다. 필터는 질문 위에서 검색
   범위를 좁혀 주는 보조 컨트롤이며, 시각적으로도 질문보다 작고 조용해야 한다.
3. **백엔드 제약을 UX로 흡수.** 백엔드 `filter_company`는 **기업명 완전 일치**로 검색한다
   (강대리 확인). 부분 명칭·오타·띄어쓰기 차이는 빈 결과가 된다. 이 제약은 사용자 잘못이
   아니라 시스템 특성이므로, **빈 결과 안내와 입력 힌트로 흡수**한다(§5.3, §3.3).
4. **한 화면, 한 흐름.** 필터·질문·결과가 위에서 아래로 한 열로 흐른다. 별도 모달·드로어를
   두지 않는다(모바일 폭에서도 동일).

## 2. 배치 — 예시 질문 영역과의 관계

### 2.1 세로 순서

```
┌─ question-card ─────────────────────────┐
│  [검색 종목  선택]        ← 필터          │
│  🔍 [ 예: 삼성전자        ✕ ]            │
│  비워 두면 전체 종목에서 검색합니다.       │
│  ─────────────────────────────  (구분선) │
│  공시에 대해 무엇이 궁금한가요?  ← 질문    │
│  [ textarea .............. ]            │
│  Enter로 질문 · Shift+Enter    [질문하기] │
└─────────────────────────────────────────┘

  이렇게 물어보세요            ← 예시 질문(examples)
  ( 칩 ) ( 칩 ) ( 칩 )
```

- **필터는 질문 카드(`question-card`) 안, 질문 입력 위**에 둔다. 검색을 실행하기 전에
  "어느 범위에서 찾을지 → 무엇을 물을지" 순으로 읽히도록 한다.
- 필터와 질문 사이에 **얇은 구분선**(`--line` 1px)을 둬서 두 컨트롤을 시각적으로 분리한다.
  필터를 카드 밖(별도 카드)으로 빼지 않는다 — 둘은 하나의 "검색 요청"을 구성하는 짝이다.

▷ 현재 구현: `company-filter`가 `question-card` 상단에 있고 `border-bottom: 1px var(--line)`으로
질문과 분리됨. 설계 의도와 일치.

### 2.2 예시 질문(examples)과의 관계 — ⚠️ 핵심 점검 포인트

예시 질문은 **초기 상태에서만** 카드 아래에 노출된다(답변·로딩·오류 시 숨김). 예시 칩을
누르면 즉시 질문이 실행된다.

여기서 **필터와 예시 질문이 충돌할 수 있다.** 현재 예시 질문 문구는 종목명을 포함한다
("삼성전자의 최근 시설투자 내용은?"). 이때 사용자가 필터에 **다른 종목**(예: "현대차")을
입력해 둔 상태로 이 예시를 누르면 → 질문은 삼성전자, 필터는 현대차 → **완전 일치 검색이
빗나가 빈 결과**가 나온다. 사용자는 이유를 알 수 없다.

**설계 결정 (택1, FE 재량 — A 권장):**

- **A. 예시 클릭 시 필터를 무시(전체 검색)한다.** 예시는 "둘러보기" 용도이므로 필터 조건 없이
  실행한다. 구현: 예시 실행 경로만 `company`를 빈 값으로 전달. 사용자가 필터를 걸어 둔 걸
  덮어쓰지는 말고(입력값은 유지), **그 요청에 한해서만** 전체 검색으로 보낸다.
- **B. 예시 문구에서 종목명을 제거**하고 필터를 실제로 쓰게 한다. 예: "최근 시설투자 내용을
  요약해줘" + 안내 문구 "위 종목 필터와 함께 눌러 보세요". 필터가 비어 있으면 전체 검색.

> A/B 중 무엇을 택하든, **"예시 문구의 종목 + 다른 필터 종목" 조합이 조용히 빈 결과가 되는
> 현재 동작은 피해야 한다.** 이게 이 필터에서 가장 흔할 혼란 지점이다.

▷ 현재 구현: A안을 적용했다. 예시 요청은 빈 `company`로 전체 검색하고 입력 중인 필터값은
유지한다.

## 3. 컨트롤 형태와 상호작용

### 3.1 컨트롤 형태 — 자유 입력(text input)

지금 단계에서는 **자유 입력 텍스트 필드**가 맞다. 이유:

- Phase 2에는 아직 사용자별 watchlist가 없다(그건 Phase 3). 선택지를 미리 제공할 종목
  마스터 목록이 프런트에 없으므로 드롭다운·칩 셀렉트는 시기상조다.
- 다만 백엔드가 완전 일치이므로 자유 입력은 오타에 취약하다 → §3.3의 입력 힌트와 §5.3의 빈
  결과 안내로 반드시 보완한다.

**구성 요소**

| 요소 | 사양 |
|---|---|
| 라벨 | "검색 종목" + 보조 배지 "선택"(muted, 11px) |
| 좌측 아이콘 | 돋보기(장식, `aria-hidden`) |
| 입력 | placeholder `예: 삼성전자`, `maxLength=100`, `autoComplete="off"` |
| 우측 지우기 버튼 | 값이 있고 로딩 중이 아닐 때만 노출, `aria-label="검색 종목 지우기"` |
| 하단 도움말 | "비워 두면 전체 종목의 공시에서 검색합니다." (muted, 11px) |

▷ 현재 구현: 위 구성과 일치.

### 3.2 상태 전이 (interaction)

- **입력 중:** 좌측 아이콘 유지, 값이 생기면 우측에 지우기(✕) 버튼 등장. 포커스 시 테두리
  `--primary` + 링 `0 0 0 3px rgba(primary,.1)`, 배경 `--surface-soft → #fff`(다크: `#111a29`).
- **지우기:** ✕ 클릭 시 입력값만 비운다. 질문·답변·결과는 건드리지 않는다. 포커스는 입력으로
  되돌리는 게 이상적(§7 접근성).
- **제출:** 질문 제출 시점의 필터값을 확정 스냅샷(`submittedCompany`)으로 잡아, 로딩·답변·재시도
  문구가 "검색 시점의 종목"을 일관되게 가리키게 한다. 제출 후 사용자가 필터를 바꿔도 방금
  결과의 표시는 흔들리지 않아야 한다.
- **제출 후 필터 유지:** 답변이 나온 뒤에도 필터 입력값은 남긴다(연속 질문 대비). 지우고
  싶으면 사용자가 ✕로 지운다.

▷ 현재 구현: `submittedCompany` 스냅샷, 지우기, 포커스 링 모두 반영됨. ✕ 클릭 후 입력으로 포커스
복귀는 미확인 → §7 참고.

### 3.3 입력 힌트 (완전 일치 대비) — ▷ 보완 권장

완전 일치 제약을 사용자가 사전에 알 수 있게, 도움말을 한 줄 강화할 수 있다.

- 현재: "비워 두면 전체 종목의 공시에서 검색합니다."
- 보완안: 위 문구 유지 + 빈 결과가 났을 때만 §5.3의 정정 안내를 노출(상시 노출은 잡음이므로
  **평시엔 조용히, 실패했을 때만 안내**한다).

정식 종목명 예시("삼성전자", "현대차", "SK하이닉스")를 placeholder 회전이나 도움말로 살짝
보여주는 것도 오타를 줄인다. 단, 과하게 설명하지 않는다.

## 4. 시각 사양 (디자인 토큰 매핑)

기존 토큰(`globals.css`)을 그대로 쓴다. 새 색/반경/그림자를 도입하지 않는다.

| 항목 | 값 / 토큰 |
|---|---|
| 입력 높이 | 44px (질문 textarea보다 낮게 — 위계 유지) |
| 입력 반경 | 11px (`company-input-wrap input`) |
| 입력 배경 | `--surface-soft` → 포커스 시 흰색/`#111a29`(dark) |
| 입력 테두리 | `--line` → 포커스 `--primary` + 링 `rgba(primary,.1)` |
| 라벨 | 13px / 750, 보조배지 11px `--text-muted` |
| 도움말 | 11px `--text-muted` |
| 아이콘 | 17px, `--text-muted`, stroke 1.8 |
| 지우기 버튼 | 28×28, 반경 7px, hover 시 `--line` 배경 |

**위계 규칙:** 필터 입력(44px)은 질문 textarea(min 118px, 15px 폰트)보다 **명확히 작고
낮아야** 한다. 필터가 질문만큼 크면 "무엇을 먼저 하는 화면인지"가 흐려진다.

▷ 현재 구현: 위 사양과 일치.

## 5. 상태별 화면

### 5.1 초기(Idle)

- 필터 비어 있음(전체 검색), ✕ 버튼 없음, 도움말 노출.
- 카드 아래 예시 질문 노출(§2.2의 충돌 처리 전제).
- 실패·성공 흔적 없음.

### 5.2 로딩(Loading)

- **필터·질문·제출 버튼 모두 잠금**(`disabled`, opacity .7, cursor not-allowed). 로딩 중 필터
  변경으로 결과와 조건이 어긋나는 걸 막는다.
- 로딩 상태 카드 문구가 **검색 범위를 그대로 반영**한다:
  - 필터 있음: "**{종목}**의 공시 원문을 확인하고 답변을 정리합니다."
  - 필터 없음: "**전체 종목의** 공시 원문을 확인하고 답변을 정리합니다."
- `role="status"`, `aria-live="polite"`로 1회 안내.

▷ 현재 구현: 잠금·범위 문구·`aria-live` 반영됨.

### 5.3 빈 결과(No sources) — ⚠️ 필터 UX의 핵심

답변 텍스트는 왔지만 참고 공시(`sources`)가 0건인 상태. 전용 `empty-card`로 답변 카드와
분리한다. **여기서 필터가 원인일 가능성을 반드시 짚어 준다.**

- 제목: "관련 공시를 찾지 못했습니다"
- 본문: 모델 답변 + **필터가 걸려 있을 때** 정정 유도 한 줄:
  - 예: "**'{종목}'** 으로 좁혀 검색했습니다. 종목명을 정확히 입력했는지 확인하거나, 필터를
    지우고 전체 공시에서 다시 검색해 보세요."
  - **바로 실행 가능한 액션**을 함께 제공하면 최상: `[필터 지우고 다시 검색]` 버튼(✕ + 재검색).
- 필터가 비어 있었다면(전체 검색인데도 0건) 종목 정정 문구는 **띄우지 않는다**(원인이 아니므로).

> 완전 일치 제약 때문에 "현대자동차" vs "현대차", "SK 하이닉스" vs "SK하이닉스" 같은 미세한
> 차이가 빈 결과를 만든다. 이 안내가 없으면 사용자는 서비스가 고장 났다고 느낀다.

▷ 현재 구현: 필터 원인 안내와 **"필터 지우고 다시 검색"** 액션을 반영했다.

### 5.4 오류(Error)

- `error-card`, `role="alert"`. "다시 시도" 버튼은 **직전 질문 + 직전 필터 조건**으로 재요청한다
  (조건이 유지돼야 사용자가 재현·재시도할 수 있다).

▷ 현재 구현: `executeQuestion(submittedQuestion, submittedCompany)`로 조건 보존됨. 일치.

### 5.5 정상 답변(Success)

- 답변 카드 상단 메타에 **검색 범위 라벨**을 노출: "AI 답변 · **{종목}**" / 필터 없으면
  "AI 답변 · **전체 종목**". 이 결과가 어떤 범위에서 나왔는지 사용자가 항상 알 수 있게 한다.

▷ 현재 구현: `AI 답변 · {submittedCompany || "전체 종목"}` 반영됨. 일치.

### 상태 요약표

| 상태 | 필터 표시 | 필터 잠금 | 필터 관련 안내 |
|---|---|---|---|
| 초기 | 입력값(비어 있으면 placeholder) | — | 도움말 "비워 두면 전체…" |
| 로딩 | 잠금 | ✅ | 로딩 문구에 검색 범위 반영 |
| 정상 | 유지 | — | 답변 메타에 "· {종목}/전체 종목" |
| 빈 결과 | 유지 | — | 정정 유도 + 필터 지우고 재검색 |
| 오류 | 유지 | — | 재시도가 직전 필터 조건 사용 |

## 6. 반응형 (모바일)

- **≤ 640px:** 필터 입력은 **가로 100% 폭**을 유지한다(질문 textarea와 동일 폭). 별도 축소·숨김
  없음. 최소 지원 폭 320px에서 좌우 스크롤이 없어야 한다.
- 터치 타깃: 지우기(✕) 버튼은 28×28이지만 실제 히트 영역이 **최소 40×40**이 되도록 패딩/여백을
  확보한다(모바일 오탭 방지). 입력 높이 44px는 터치 기준을 만족.
- 필터-질문 구분선 상하 여백은 모바일에서 소폭 축소(현재 17px)해 세로 공간을 아낀다.
- placeholder "예: 삼성전자"가 좁은 폭에서 잘리지 않는지 확인(잘리면 "삼성전자" 정도로 축약).

▷ 현재 구현: 100% 폭, `@media (max-width:640px)`에서 `company-filter` 여백 축소 반영됨.
지우기 버튼 히트 영역 40×40 확보는 **실측 확인 필요**.

## 7. 다크모드

- 별도 색을 새로 만들지 않고 `prefers-color-scheme: dark`의 기존 토큰 오버라이드를 따른다.
- 점검 포인트(대비):
  - 입력 배경 `--surface-soft(#1a2435)` 위 placeholder `#7f8b9d` — **본문 대비 충분한지 확인**.
  - 포커스 시 입력 배경이 `#111a29`로 바뀌며 테두리 `--primary(#78a7ff)` + 링이 보이는지.
  - 도움말/보조배지 `--text-muted(#8e9bad)`가 카드 배경 위에서 읽히는지.
  - 지우기 버튼 hover 배경 `--line(#2b3749)`이 아이콘과 구분되는지.

▷ 현재 구현: 다크 포커스 배경·placeholder 색 오버라이드 존재. **실제 OS 전환 실측으로 대비
확인 필요**(QA 체크리스트 §9).

## 8. 접근성

| 항목 | 요구 사항 | 현재 |
|---|---|---|
| 라벨 연결 | `<label for="company">` ↔ `input#company` | ✅ |
| 도움말 연결 | `input aria-describedby="company-help"` | ✅ |
| "선택" 표시 | 시각 배지로 제공. 스크린리더에도 "선택"이 라벨의 일부로 읽힘 | ✅ (라벨 내 span) |
| 지우기 버튼 | 아이콘 버튼에 `aria-label="검색 종목 지우기"`, ✕는 `aria-hidden` | ✅ |
| 포커스 표시 | `:focus-visible` 3px `--primary` 아웃라인 | ✅ (전역 규칙) |
| 포커스 이동 | ✕로 지운 뒤 **입력으로 포커스 복귀**(버튼이 사라지므로 포커스 유실 방지) | ✅ |
| 탭 순서 | 필터 입력 → (지우기) → 질문 textarea → 제출. 시각 순서와 일치 | ✅ |
| 로딩/오류/빈결과 안내 | `role`/`aria-live`로 **1회만** 안내 | ✅ |
| 상태 안내에 종목 포함 | 로딩·답변 메타에 검색 종목이 텍스트로 노출돼 보조기기로 전달 | ✅ |

**보완 1 — 포커스 복귀:** 지우기 버튼은 값이 비면 DOM에서 사라진다. 클릭 후 포커스가 body로
튀지 않게 `input#company`로 명시적으로 포커스를 돌려준다.

**보완 2 — 빈 결과 정정 액션의 접근성:** §5.3의 "필터 지우고 다시 검색" 버튼을 추가한다면,
빈 결과 카드가 `role="status"`/`aria-live`로 이미 읽히므로 버튼 라벨은 동작을 그대로 서술한다
("필터를 지우고 전체 공시에서 다시 검색").

## 9. FE 정합성 체크리스트

이 문서를 기준으로 구현이 맞는지 확인한다. `[x]`는 현재 코드로 충족, `[ ]`는 보완/실측 대상.

**배치·상호작용**
- [x] 필터가 질문 카드 상단, 질문 위에 있고 구분선으로 분리됨
- [x] 필터는 선택적이며 비우면 전체 검색
- [x] 값이 있고 비로딩일 때만 ✕ 노출, 클릭 시 입력만 비움
- [x] 예시 질문 클릭 시 해당 요청만 전체 검색하고 필터 입력값은 유지
- [x] 제출 시점 필터를 스냅샷(`submittedCompany`)으로 고정

**상태**
- [x] 로딩 중 필터·질문·제출 잠금
- [x] 로딩·답변 메타에 검색 범위(종목/전체) 반영
- [x] 빈 결과에서 필터 원인 정정 안내 + 필터 지우고 재검색 액션
- [x] 오류 재시도가 직전 질문·필터 조건 사용

**반응형·다크·접근성 (실측)**
- [x] 필터 100% 폭, 320px 가로 스크롤 없음(정적)
- [ ] 지우기 버튼 터치 히트 영역 ≥ 40×40 실측
- [ ] 다크모드 placeholder/도움말/포커스 대비 실측
- [x] ✕ 클릭 후 입력 포커스 복귀
- [x] 라벨·도움말·버튼 aria 연결

## 10. 향후 (Phase 3 확장 메모)

지금은 자유 입력이 맞지만, Phase 3에서 사용자별 **watchlist**가 생기면 이 필터는 자연스럽게
**콤보박스(입력 + 내 관심종목 제안)**로 진화한다.

- 입력하면 내 watchlist / 최근 검색 종목을 제안(datalist 또는 커스텀 listbox).
- 완전 일치 오타 문제가 **선택으로 대체**되어 사라진다 → §5.3 정정 안내의 필요성도 감소.
- 접근성은 `role="combobox"` + `aria-expanded`/`aria-activedescendant` 패턴을 따른다.

이 진화 경로를 염두에 두고, 현재 구현의 라벨/도움말/상태 문구는 "종목을 좁혀 검색한다"는
멘탈 모델을 유지한다(나중에 컨트롤만 바뀌고 모델은 그대로여야 이관 비용이 적다).
